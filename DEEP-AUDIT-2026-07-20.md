# Metals Encyclopedia — Deep Audit Report

**Date:** 2026-07-20  
**Method:** 10 parallel finder agents (5 Opus 4.8 max + 5 Sonnet 5 max) across distinct dimensions → independent adversarial verification of every medium-and-above finding (cross-model, prompted to refute) → Opus 4.8 synthesis.  
**Fleet stats:** 56 agents · 3,537,890 subagent tokens · 970 tool calls · ~39 min.

## Result

- Overall health: **FAIR**
- Findings: **71 raw → 71 unique → 69 final** (after verification)
- Verification: 45 checked, 43 survived, **2 refuted**
- Severity: 🔴 0 critical · 🟠 4 high · 🟡 26 medium · ⚪ 39 low

## Executive summary

This is a static scientific reference encyclopedia, so its entire value proposition is trustworthy data — and that is exactly where the audit finds the most damage. The app is solid as an artifact: it builds, deploys, and renders; the large majority of the ~thousands of numeric data points across 98 elements are correct; there is no actively exploitable security hole (all current data resolves clean, images are same-origin, and the XSS sinks are latent); escapeHtml exists and is used in most places; and the defects are well-localized and fixable rather than architectural rot. The provenance scaffolding and audit scripts show genuine intent toward quality.

What's risky cuts to the core promise. (1) Flagship accuracy failures: gold's Brinell hardness is ~10x too high (2,450 MPa — as hard as uranium), silver's is ~10x too low (24.5 MPa — softer than lead), a mirror-image x10/÷10 scaling bug on the two marquee precious metals, and germanium carries a physically impossible 7,273 MPa fabricated to 13 significant figures — all rendered on the user-facing detail sheet. (2) The trust architecture is hollow: the 'Confirmed' badge marks source provenance, not per-claim verification, is computed from mere excerpt-extractability rather than coverage, and is stamped on demonstrably false prose (gold production geography, aluminum ores as 'silicates', Chile as lithium's top producer) and on synthetic elements (promethium, scandium) cited to a Tier-1 USGS rare-earths chapter that explicitly excludes them, complete with 'China 70%' import statistics that describe nothing about them. (3) The pipeline has no drift guard — the last committed build shipped with commodityNotes missing on all 98 elements (verified: HEAD had 0, the generator has 98), so npm run dev silently serves stale content. (4) Accessibility was never baselined: no h1 on any core content page, no live regions anywhere, contrast fails WCAG on every data label, and the core interactions (Compare reorder, the Add-metal dialog, custom dropdowns) are keyboard-inaccessible. (5) On-screen self-contradiction: Bohr shell counts are wrong for every element from Z=19 up and disagree with the accurate electron configuration shown beside them. Performance (full innerHTML re-render per click, ~11.6MB of un-resized megapixel images eagerly preloaded for ~100px thumbnails) and inconsistent escaping/no-CSP are real but rank below correctness, trust, and a11y for this product. Net: a workable foundation carrying serious, user-visible accuracy and trust defects that undercut the one thing an encyclopedia must get right.

## Cross-cutting themes

- Confidence signals outrun verification — the 'Confirmed' badge, Tier-1 citations, and present-tense statistics assert more certainty than the content has earned, and several demonstrably false or synthetic-element claims ship under them.
- No pipeline guardrails — generated artifacts drift from their source (metals.js shipped missing all commodityNotes), --apply doesn't rebuild as documented, audit scripts are stale/false-positive (missing uranium, arsenic sublimation), and auto-patches lack sanity bounds.
- Accessibility was never baselined — failures span page structure (no h1), contrast, live regions, keyboard operability, focus management, and touch-target size across the entire app rather than in isolated spots.
- A small set of egregious, user-facing numeric errors amid mostly-correct data — gold/silver hardness (mirror-image scaling bug), germanium's fabricated value, plus Bohr shell counts that contradict the electron config on the same screen.
- Defensive rendering is applied inconsistently — escapeHtml is used in most sinks but skipped in img src, four href attributes, and desktop compare options, with no URL-scheme allow-listing and no CSP backstop; latent today because current data is clean, not by design.
- Rebuild-the-world rendering and heavy assets — full innerHTML re-render (including the 98-card sidebar) on every interaction, ~11.6MB of un-resized images eagerly preloaded for tiny thumbnails, and no debounce or memoization anywhere.
- Dead and reserved surface accumulating unread weight — an inert viewer stub, an auto-generated unused 'summary' field, byte-duplicated propertySources, and dead/redundant CSS all carried in the shipped output with zero runtime consumers.

## Top priorities (ranked by impact ÷ effort)

**1. Correct the gold, silver, and germanium hardness values and hunt the scaling bug behind them** _(effort: trivial)_  
Two marquee precious metals show egregiously wrong, user-visible numbers on the detail sheet — gold Brinell 2,450 MPa (hard as uranium), silver 24.5 MPa (softer than lead in the same dataset). The exact mirror-image x10/÷10 pattern points to a scaling bug that may have other victims worth sweeping for, and germanium's 7,273 MPa is a fabricated, physically impossible value. This is the cheapest, most visible correctness win in the whole audit.

**2. Rebuild metals.js and add a CI/pre-commit guard so the generated file can't drift from its inputs** _(effort: small)_  
The committed build shipped with commodityNotes missing on all 98 elements (verified: HEAD=0, narratives.mjs=98), so the USGS commodity-notes block renders empty and the normal npm run dev workflow serves stale narratives whenever generators change. A check that regenerates and diffs metals.js — plus fixing import-vault-numerics --apply, which its own docstring says rebuilds but doesn't — stops silent staleness from recurring.

**3. Ship the cheap app-wide accessibility fixes: darken --faint, add a page <h1>, add a live region** _(effort: small)_  
Three small changes with whole-app reach: the --faint token fails WCAG contrast on every small-caps data label on all 98 element pages plus Compare and Table (one CSS token); the app's core content (every element page, Compare, Table) has no h1 for AT users to orient on; and search-result counts and Compare notices are silent to screen readers because no aria-live/role=status exists anywhere.

**4. Correct the false/outdated flagship narratives and fix what the 'Confirmed' badge actually asserts** _(effort: medium)_  
Credibility is the product. Gold's production geography, aluminum ores classified as 'silicates', Chile as lithium's top producer, manganese(II) oxide as an oxidiser, and flerovium 'formed in nuclear reactors' all ship as present-tense fact under a 'Confirmed' badge that only tracks provenance, not verification. Either rename the badge to reflect what it means or actually fact-check the claims it vouches for.

**5. Repair citation integrity for synthetic and mis-mapped elements and gate confidence on real coverage** _(effort: medium)_  
Promethium and scandium are stamped 'Confirmed' with Tier-1 USGS rare-earths citations and 'China 70%' import stats that the cited chapter explicitly says exclude them — describing nothing about a synthetic element that isn't a mined commodity. The confidence engine grants 'Confirmed' on excerpt-extractability from a mapped chapter, not on whether that chapter documents the element, inflating the user-facing 'Confirmed' tally. A coverage check plus removing inapplicable commodityNotes fixes a genuine credibility landmine.

**6. Fix the Bohr-model shell counts (wrong for every element Z>=19)** _(effort: medium)_  
shellCounts() fills shells to 2n^2 maxima, which only matches reality through argon; from potassium up it ignores the 4s-before-3d ordering, so the animated rings and 'BOHR MODEL · SHELLS' caption are wrong for most of the table and visibly contradict the accurate electronConfiguration string rendered on the same data sheet — one screen showing two conflicting numbers.

**7. Resize the sample-image corpus and debounce search inputs** _(effort: medium)_  
public/samples ships ~11.6MB of un-resized megapixel photos (actinium.png alone is 1.37MB) rendered into ~100px thumbnails, and warmMetalImages eagerly preloads up to 48 full-resolution files on every browse/filter action (~6MB per interaction) bypassing loading=lazy — a mechanical, high-impact fix via a resize step plus srcset. Adding debounce to the search inputs (none exists anywhere) stops rebuilding the 98-element search haystack on every keystroke.

**8. Close the keyboard and focus accessibility gaps and size touch targets** _(effort: large)_  
The interactive core is unusable without a mouse: Compare column reorder (which sets the baseline every delta is computed against) has no keyboard path, the Add-metal dialog is aria-modal but never receives focus and ignores Escape with no focus trap, the custom Sort/Series listboxes don't close on Escape and give every option its own tab stop, and the mobile remove button is 18x18px (under the WCAG 2.2 24px minimum) while sharing its surface with a drag gesture.

### 🟠 1. Gold overview states grossly outdated production volume and geography as current fact
- **Where:** `scripts/narratives.mjs:1917`
- **Dimension:** Narrative factuality & hallucination (scripts/narratives.mjs prose) · finder: Opus 4.8 · **✅ confirmed**
- **What:** The gold overview presents 1970s/80s-era mining statistics in the present tense on the encyclopedia's flagship precious metal. Both the annual tonnage and the country attribution are wrong by a wide margin today.

### 🟠 2. Aluminum overview falsely classifies bauxite and cryolite as 'aluminium silicates'
- **Where:** `scripts/narratives.mjs:159`
- **Dimension:** Narrative factuality & hallucination (scripts/narratives.mjs prose) · finder: Opus 4.8 · **✅ confirmed**
- **What:** A chemically false mineral classification on a flagship element. Neither named ore is a silicate.
- **Evidence:** Repo (aluminum.overview): "It is usually found in minerals such as bauxite and cryolite. These minerals are aluminium silicates." Bauxite is a mixture of hydrated aluminium oxides/hydroxides (gibbsite Al(OH)3, boehmite/diaspore AlO(OH)); cryolite is sodium aluminium fluoride, Na3AlF6. Neither contains silicon. ### 🟠 3. Gold brinellHardness is ~10x too high (2450 MPa vs 188-245 MPa)
- **Where:** `src/data/metals-raw.json:1692`
- **Dimension:** Scientific numeric accuracy (physical & chemical properties, all 98 elements) · finder: Opus 4.8 · **✅ confirmed**
- **What:** Gold's brinellHardness = 2450 MPa makes the softest coinage metal appear as hard as uranium (U=2400 MPa here) or hardened steel. Rendered to users on the detail sheet as "Brinell 2,450.00 MPa" (main.js:896).
- **Evidence:** Repo: "brinellHardness": 2450 (metals-raw.json:1692). Reference: Wikipedia gold infobox Brinell hardness = "188-245 MPa"; pure annealed gold is ~25 HB (kgf/mm2) = ~245 MPa. The repo value is exactly 10x the 245 upper bound.
- **Fix:** Correct to ~245 MPa (the accepted 188-245 range).

### 🟠 4. Silver brinellHardness is ~10x too low (24.5 MPa vs 206-250 MPa), making it softer than lead
- **Where:** `src/data/metals-raw.json:882`
- **Dimension:** Scientific numeric accuracy (physical & chemical properties, all 98 elements) · finder: Opus 4.8 · **✅ confirmed**
- **What:** Silver's brinellHardness = 24.5 MPa makes silver appear SOFTER than lead (Pb = 38.3 MPa in this same dataset), which is physically wrong: silver is substantially harder than lead. This is the mirror image of the gold error (gold got x10, silver got /10), pointing to a scaling bug on these two records specifically. Rendered as "Brinell 24.50 MPa" (main.js:896).
- **Evidence:** Repo: "brinellHardness": 24.5 (metals-raw.json:882). Reference: Wikipedia silver infobox Brinell hardness = "206-250 MPa". The repo value equals ~245/10. For comparison the same file has lead=38.3 MPa (correct) and copper=874 MPa (correct), so silver at 24.5 is internally inconsistent.
- **Fix:** Correct to ~245 MPa (accepted 206-250 range).

## 🟡 MEDIUM (26)

### 🟡 1. Compare column reordering has no keyboard equivalent (desktop drag-and-drop, mobile pointer-drag)
- **Where:** `src/main.js:1394`
- **Dimension:** Accessibility (a11y) · finder: Sonnet 5 · **✅ confirmed**
- **What:** The Compare tool's reorder gesture — which determines the 'baseline' column that every delta/percentage in the matrix is computed against — is implemented purely as native HTML5 drag-and-drop on desktop (dragstart/dragover/drop on a plain draggable div) and raw Pointer Events on mobile (pointerdown/pointermove/pointerup on a plain div). Neither has a keydown/button alternative. A whole-file search of src/main.js turns up exactly one keydown listener in the entire app (the °C/°F toggle at line 1787) — none for reordering. Live-checking the actual rendered mobile drag handle confirms it isn't even reachable by keyboard: it's a bare <div> with no tabindex and no role, so Tab skips over it entirely. A keyboard user can approximate a reorder only by removing every metal after their desired baseline and re-adding them (toggleCompareKey always appends at the end), which is not discoverable, not equivalent, and destroys/rebuilds the set rather than moving one item.
- **Evidence:** Desktop: `<div class="compare-card${selectedCls}" draggable="true" data-compare-key="${m.key}">` (src/main.js:1394), wired only via `card.addEventListener("dragstart", ...)` / `cardsRoot?.addEventListener("dragover"/"drop", ...)` (src/main.js:1821-1862). Mobile: `<div class="compare-sheet-head" data-drag-head="${m.key}" title="Drag to reorder">` (src/main.js:1325), wired via `head.addEventListener("pointerdown"/"pointermove"/"pointerup", ...)` (src/main.js:1691-1726). Live DOM check of the rendered handle: `{"headTag":"DIV","headTabindex":null,"headRole":null}`. WCAG 2.1.1 (Level A, Keyboard) requires all functionality operable through a keyboard interface; none is provided here.
- **Fix:** Add explicit 'Move left/right' (or up/down) buttons per compare column, or make the drag handle a real [tabindex="0"][role="button"] with ArrowLeft/ArrowRight keydown handlers that call the already-existing reorderCompareKeys() helper from src/lib/compare.js.
- **Verifier correction:** Defect is real as described. Only refinement: it is a scoped Level A failure on the reorder/baseline gesture, not a total keyboard lockout of the Compare tool — add, remove, select, and preset controls are keyboard-accessible buttons.

### 🟡 2. "Add metal" dialog (role=dialog, aria-modal=true) never receives focus and ignores Escape
- **Where:** `src/main.js:1373`
- **Dimension:** Accessibility (a11y) · finder: Sonnet 5 · **✅ confirmed**
- **What:** The mobile Compare 'Add metal…' bottom sheet is marked up as a modal dialog but has none of the behavior that role implies: opening it does not move focus into it, closing it is only possible via a backdrop click, and Escape does nothing. ### 🟡 5. No live region anywhere in the app — search-result counts and Compare notices are silent to screen readers
- **Where:** `src/main.js:838`
- **Dimension:** Accessibility (a11y) · finder: Sonnet 5 · **✅ confirmed**
- **What:** Not one element in the whole app carries aria-live, role="status", or role="alert" (confirmed both by grep of src/main.js and by a live query against the running DOM: 0 matches for all three). The dominant, best-evidenced finding is that the entire production JS bundle is a single 504.44 kB (101.86 kB gzip) chunk — confirmed by an actual build, with Vite's own build warning flagging it — of which measurement shows ~89% by byte-weight is the 98-element narrative dataset (metals.js), none of it code-split, all of it parsed and held in memory before any element is even opened. Several concrete, fixable sources of avoidable weight/re-render cost stack on top of that (duplicated static sub-objects baked into all 98 records, un-debounced search that reprocesses ~143K characters of narrative text per keystroke, full-shell + full-98-card-list innerHTML rebuilds on every single element click, and full-resolution multi-hundred-KB images with no responsive sizing eagerly bulk-preloaded in a way that defeats the `loading="lazy"` attribute set on the same elements).
